P1 (0,5p). Creación y Actualización de Vistas

Enunciado:
Considera las vistas vista1 y vista2 a continuación (se asume que la vista emps_depto10 ya existe y es correcta). Para cada una de ellas, indica si la vista se crea o si la sentencia de creación da un error (en este caso, di por qué). En el caso de que se cree, indica si es actualizable y si permitiría la inserción de filas migratorias. Justifica las respuestas.

Definición de vistas:

SQL
CREATE VIEW vista1 AS
SELECT DISTINCT empno, ename, sal+comm, hiredate
FROM emp
WHERE deptno = 30;


(Vista base correcta previa):
SQL
CREATE VIEW emps_depto10 AS 
SELECT * FROM emp WHERE deptno = 10;


SQL
CREATE VIEW vista2 AS
SELECT empno, ename, sal, job
FROM emps_depto10
WHERE sal > 4000
WITH LOCAL CHECK OPTION;
🎯 Resolución:
1. vista1: Se crea sin problemas. No es actualizable ya que tiene una cláusula de filtrado de repetidos (DISTINCT). Por otra parte, sí que permite la inserción de filas migratorias porque no tiene la opción de CHECK OPTION especificada en su definición.


2. vista2: Se crearía sin problema. Sería actualizable y no existirían filas migratorias de forma local. Sin embargo, si insertásemos, por ejemplo, un empleado con un sueldo superior a 4000 pero perteneciente al departamento 20, la vista lo permitiría (ya que la restricción local solo comprueba el salario), originando una fila migratoria respecto a la vista base emps_depto10


P2 (0,5p). Seguridad y Revocación de Roles

Enunciado:
Di qué pasaría si un usuario con el rol ABD ejecutase la siguiente sentencia SQL en Oracle. Específicamente, ¿qué ocurre con los usuarios que ya están conectados?

SQL
REVOKE CONNECT FROM PUBLIC;
🎯 Resolución:
Causaría la revocación del rol CONNECT (el cual posee el privilegio CREATE SESSION necesario para abrir una sesión contra la base de datos) a absolutamente todos los usuarios, dado que PUBLIC es un pseudo-AuthID que engloba a todos los usuarios del sistema (presentes y futuros).

Este efecto no sería instantáneo para los usuarios ya conectados, dado que la revocación de un rol no se hace efectiva hasta que se inicia una nueva sesión (CONNECT usr/key) o se ejecuta explícitamente un SET ROLE ALL. Afectará, por lo tanto, a las sesiones futuras y a las presentes en cuanto intenten revalidar sus roles.

P3 (0,5p). Optimización y Uso de Índices

Enunciado:
Considera la tabla EMP de docencia, que tiene la restricción PK_EMP (clave primaria sobre EMPNO). Además, se definió un índice I_EMP_COMM sobre el campo COMM, y un índice I_EMP_SAL sobre el campo SAL. Di si para ejecutar las siguientes sentencias Oracle utilizaría algún índice (en caso afirmativo, di el nombre del índice). Justifica la respuesta.

SQL
SELECT /*+ INDEX (EMP I_EMP_COMM) */ DISTINCT (COMM)
FROM EMP;
🎯 Resolución:
• Para la primera sentencia: El DISTINCT lo que hace es agrupar, por lo que los valores nulos se agrupan en una única fila. En Oracle, los valores completamente nulos no se incluyen en los índices B-Tree normales, por lo que no se utilizará el índice (aunque esté la sugerencia/hint).
SQL
SELECT AVG(SAL)
FROM EMP;


[!success]- Solución
• Para la segunda sentencia: El campo SAL puede contener valores nulos, pero como se está realizando una operación de agrupamiento (AVG), Oracle sí va a usar el índice I_EMP_SAL para optimizar el cálculo de agregación sobre los valores no nulos.

P5 (0,75p). Control de Concurrencia Multiversión

Enunciado:
Considera un sistema con control de concurrencia multiversión. La marca de transacción será su subíndice, esto es, para cada transacción $T_i$, $M(T_i) = i$.

Para el dato A tenemos inicialmente las siguientes versiones:

$A_1 = 20$$\text{ ML} = 4$
$\text{ME} = 1$
$A_2 = 30$$\text{ ML} = 7$
$\text{ME} = 6$
Indica lo que pasa en cada una de las siguientes operaciones secuenciales y cómo quedan las versiones de A:

• $T_3$ lee A.
🎯 Resolución:
Efecto: Como $M(T_3) = 3$, busca la versión más reciente cuyo $\text{ME} \le 3$, que es $A_1$ ($\text{ME}=1$). Dado que $M(T_3) = 3$ no supera el $\text{ML}$ actual ($\text{ML}=4$), $T_3$ lee el valor $A_1 = 20$ sin modificar las marcas.

Estado de versiones:

• $A_1 = 20 → \text{ME} = 1, \text{ML} = 4$
• $A_2 = 30 → \text{ME} = 6, \text{ML} = 7$


• $T_8$ lee A.


[!success]- Resolución

Efecto: Como $M(T_8) = 8$, la versión adecuada es $A_2$ ($\text{ME}=6 \le 8$). Como $M(T_8) = 8 > \text{ML}(A_2) = 7$, se actualiza la marca de lectura $\text{ML}(A_2) = 8$. $T_8$ lee el valor $A_2 = 30$.

Estado de versiones:

• $A_1 = 20 → \text{ME} = 1, \text{ML} = 4$
• $A_2 = 30 → \text{ME} = 6, \text{ML} = 8$


• $T_9$ escribe el valor 13 en A.


[!success]- Resolución

Efecto: Como $M(T_9) = 9$ y es mayor que el $\text{ML}$ de la versión más reciente ($\text{ML}(A_2) = 8$), se permite la escritura. $T_9$ crea una nueva versión $A_3 = 13$.

Estado de versiones:

• $A_1 = 20 → \text{ME} = 1, \text{ML} = 4$
• $A_2 = 30 → \text{ME} = 6, \text{ML} = 8$
• $A_3 = 13 → \text{ME} = 9, \text{ML} = 9$


• $T_3$ escribe el valor 25 en A.


[!success]- Resolución

Efecto: $T_3$ intenta acceder a $A_1$ ($\text{ME}=1$). Por el contrario, como su marca $M(T_3) = 3$ es menor que alguna lectura ya realizada por otras transacciones posteriores ($\text{ML}=4$ o $\text{ML}=8$), la operación se deniega para evitar anomalías. $T_3$ hace rollback y se reinicia con una nueva marca temporal.

• $T_5$ escribe el valor 65 en A.


[!success]- Resolución

Efecto: Al realizarse tras los cambios y las reorganizaciones de versiones del ejercicio, se deteriora la secuencia y se registra la creación de una versión intermedia válida o reajuste pendiente. (El alumno dibuja la inserción ordenando las versiones resultantes por marcas de escritura: $A_1$, nueva versión $A_{\text{intermedia}}$ con $\text{ME}=5, \text{ML}=5$, seguida de $A_2$ y $A_3$).

Tras la última entrada del log se produjo una caída sin pérdida de almacenamiento no volátil.

Di si el log corresponde a un sistema con actualizaciones inmediatas o diferidas (y por qué lo sabes).

Indica lo que habría que hacer para recuperar el sistema ante esa caída (suponiendo que se utilizan las estrategias Robar y No forzar).
🎯 Resolución:
Tipo de sistema: Corresponde con un sistema de actualizaciones inmediatas. Lo sabemos porque en las entradas de modificación del log se guardan concurrentemente tanto el valor anterior (imagen antigua) como el valor posterior (imagen nueva), necesarios para poder hacer tanto undo como redo.

Operaciones de recuperación:

• Se ignora T1: Su COMMIT ocurrió antes del , lo que asegura que sus cambios ya están volcados a disco de forma estable y no es necesario procesarla.
• redo(T2): Estaba activa en el momento del checkpoint y consolidó con su correspondiente COMMIT antes de la caída.
• undo(T3): Comenzó su ejecución pero no llegó a consolidar con su correspondiente COMMIT antes de la caída.

P6b (0,25p). Restricciones Aplazables (Deferrable)

Enunciado:
¿Qué es una restrición aplazable? Pon un ejemplo donde sea necesario su uso.
🎯 Resolución:
Las restricciones aplazadas son aquellas definidas con la opción DEFERRABLE. Al contrario que las normales (NOT DEFERRABLE), que se comprueban de manera inmediata tras lanzar cada query, con las aplazables podemos modificar el comportamiento con la sentencia:

SQL
SET CONSTRAINTS { ALL | restr1, ... } { IMMEDIATE | DEFERRED };


• IMMEDIATE: Se comporta en el momento, de manera inmediata.
• DEFERRED: Esperará hasta el momento de confirmar la transacción (COMMIT) para comprobar la validez de la integridad.


Ejemplo de uso: Es totalmente necesario para romper ciclos referenciales entre dos tablas (por ejemplo, cuando la tabla A tiene una foreign key no nula hacia la tabla B, y la tabla B tiene otra foreign key no nula hacia la tabla A). Sin aplazar la restricción, sería imposible insertar el primer registro en ninguna de ellas.

a) Escribe una sentencia SQL que pueda generar ese plan de ejecución.
🎯 Resolución:
SQL
SELECT e.deptno, d.dname, COUNT(*)
FROM emp e JOIN dept d ON e.deptno = d.deptno
GROUP BY e.deptno, d.dname
HAVING COUNT(*) > 3;

Primero se realizan las operaciones 5 y 7. La 5 hace una exploración completa del índice (INDEX FULL SCAN de PK_DEPT) en la tabla de departamentos, y la 7 realiza un escaneo completo (TABLE ACCESS FULL) de la tabla EMP para leer todas sus filas.

Después, se ejecuta la operación 6 que ordena (SORT JOIN) las filas obtenidas de EMP por la columna de unión (DEPTNO). Paralelamente, la 4 accede por ROWID a las filas necesarias asociadas al índice de departamentos. La operación 3 mezcla las dos fuentes mediante un MERGE JOIN.

Finalmente, la operación 2 agrupa los resultados combinados (HASH GROUP BY) por departamento. La operación 1 aplica el filtro del HAVING (COUNT(*) > 3) para descartar los grupos con 3 o menos filas, y la 0 devuelve el resultado final procesado por la SELECT STATEMENT.
