P1. Vistas y Modificaciones de Esquema (SQL)

Enunciado:
Considera las siguientes sentencias ejecutadas en orden secuencial por los usuarios indicados entre paréntesis:

1.
SQL
(u1) CREATE VIEW vista1 AS SELECT * FROM dept WHERE deptno = 30;
2.
SQL
(u1) GRANT SELECT, INSERT ON vista1 TO u2;
3.
SQL
(u1) ALTER TABLE dept ADD phone VARCHAR(20);
4.
SQL
(u2) SELECT * FROM u1.vista1;
5.
SQL
(u2) INSERT INTO u1.vista1 VALUES (50, 'I+D', 'BOSTON');


¿Qué ocurre exactamente en las sentencias 4 y 5?
🎯 Resolución:
• Sentencia 4: Cuando u2 ejecuta la selección, se mostrarán todas las columnas originales de la tabla dept que cumplan la condición, exceptuando la nueva columna phone. Esto ocurre porque la vista es un objeto precompilado que solo conoce los atributos de la tabla en el momento exacto en el que fue creada.
• Sentencia 5: Aunque u2 tiene privilegios de inserción, la sentencia producirá un error de integridad o de validación (dependiendo del SGBD). El valor 50 para deptno viola la condición implícita de la vista (deptno = 30). Si la vista se hubiese creado WITH CHECK OPTION, el SGBD lo rechazaría explícitamente; si no, la fila podría insertarse en la tabla base pero "desaparecería" inmediatamente de la vista. Además, al mapear un INSERT genérico (VALUES) sin especificar columnas sobre una tabla modificada estructuralmente, la discrepancia en el número de atributos provocará un fallo de compilación de la consulta.

P2. Gestión de Privilegios y Pseudo-roles

Enunciado:
Si el usuario admin tiene el rol global de DBA, indica la validez y el efecto de las siguientes acciones independientes:

SQL
a) (admin) REVOKE PUBLIC FROM Usuario;
🎯 Resolución:
Respuesta a): No es posible ejecutarlo. PUBLIC no es un rol otorgable ni revocable individualmente; es un pseudo-rol o identificador de autorización especial que representa automáticamente a absolutamente todos los usuarios de la base de datos. ``` sql b) (admin) GRANT CONNECT TO usuario WITH GRANT OPTION; (usuario) REVOKE CONNECT FROM admin; ``` [!success]- Solución Respuesta b): La primera sentencia funciona sin problemas: admin otorga el privilegio/rol CONNECT a usuario confiriéndole además la facultad (WITH GRANT OPTION) de poder extender ese mismo permiso a terceros. Sin embargo, la segunda sentencia dará un error y no será posible: un usuario común no puede revocar privilegios sobre un administrador si él no fue quien se los concedió originalmente.

P3. Control de Concurrencia por Marcas de Tiempo (Thomas Write Rule)

Enunciado:
Simula el comportamiento del siguiente plan de ejecución aplicando la Regla de Escritura de Thomas. Si en el proceso fuera necesario anular alguna transacción por violación de integridad, ignora todas sus operaciones futuras.

• Estado inicial: Las marcas de lectura (ML) y escritura (ME) para los ítems A, B y C son 0. Cada transacción $T_i$ opera con la marca de tiempo de su propio subíndice ($M(T_i) = i$).


InstanteT1​T2​T3​
1Read(A)
2Write(B)
3Write(B)
4Read(A)
5Read(C)
6Read(A)
7Read(B)
8Write(C)
🎯 Resolución:
1. $T_1$: Read(A) $→$ Posible. Como $M(T_1) = 1 \ge ME(A) = 0$, la lectura es válida.
• Estado: ML(A) = 1, ME(A) = 0.
2. $T_3$: Write(B) $→$ Posible. Como $M(T_3) = 3 \ge ML(B) = 0$ y $\ge ME(B) = 0$, se escribe el dato.
• Estado: ML(B) = 0, ME(B) = 3.
3. $T_1$: Write(B) $→$ Se ignora por Regla de Thomas. Como $M(T_1) = 1 < ME(B) = 3$, se está intentando escribir un valor más antiguo que el ya consolidado por $T_3$. La transacción $T_1$ no aborta, simplemente omite la escritura.
• Estado: ML(B) = 0, ME(B) = 3.
4. $T_3$: Read(A) $→$ Posible. Como $M(T_3) = 3 \ge ME(A) = 0$, la lectura es válida.
• Estado: ML(A) = 3, ME(A) = 0.
5. $T_3$: Read(C) $→$ Posible. Como $M(T_3) = 3 \ge ME(C) = 0$.
• Estado: ML(C) = 3, ME(C) = 0.
6. $T_2$: Read(A) $→$ Posible. Como $M(T_2) = 2 \ge ME(A) = 0$. La marca de lectura no se modifica porque $ML(A)=3$ ya es mayor.
• Estado: ML(A) = 3, ME(A) = 0.
7. $T_2$: Read(B) $→$ FALLO (Rollback de $T_2$). Como $M(T_2) = 2 < ME(B) = 3$, la transacción $T_2$ intenta leer una versión de B modificada por una transacción más nueva ($T_3$). $T_2$ se aborta y se reiniciará.
8. $T_1$: Write(C) $→$ FALLO (Rollback de $T_1$). Como $M(T_1) = 1 < ML(C) = 3$, $T_1$ intenta escribir un registro que ya ha sido leído por una transacción más reciente ($T_3$). $T_1$ se aborta y se reiniciará.

P4. Acciones Referenciales (`ON DELETE / ON UPDATE NO ACTION`)

Enunciado:
Explica el funcionamiento y la semántica de la acción referencial NO ACTION. Puedes ayudarte de un ejemplo para completar tu explicación.
🎯 Resolución:

Respondido por el alumno:



La cláusula NO ACTION especifica que si los cambios que se pretenden realizar sobre una clave primaria o única violan o rompen la integridad referencial, la operación de borrado o actualización no se ejecutará y el motor de la base de datos devolverá un error. A diferencia de RESTRICT, NO ACTION permite que la verificación se pueda aplazar hasta el final de la transacción (DEFERRED) si la restricción está configurada con un comportamiento aplazable.

* Ejemplo: Imaginemos las tablas relativas a empleados (emp) y departamentos (dept), donde la columna deptno en emp es una clave foránea que apunta a dept. Si en la tabla dept tenemos una tupla indicando que hay un departamento cuyo número es 10 y un empleado que está asignado a dicho departamento en emp, cualquier intento de eliminar el departamento 10 o de modificar su número abortará devolviendo un error de integridad referencial, ya que existen registros hijos que todavía dependen de él.

P5. SQL y Álgebra Relacional (Forma Canónica y Optimizada)

Enunciado:
Considerando el siguiente esquema relacional de música:

erDiagram ARTISTA ||--o{ COMPON : "compone" CANCION ||--o{ COMPON : "se compone en" ARTISTA { string IDART PK string NOME int ANONAC } COMPON { string IDART PK, FK string IDCAN PK, FK } CANCION { string IDCAN PK string TITULO string ESTILO int DURACION }
🎯 Resolución:
SQL
SELECT a.nome, SUM(can.duración)
FROM artista a, compone c, canción can
WHERE a.idart = c.idart 
	AND c.idcan = can.idcan 
	AND a.anonac > '2000' 
	AND can.estilo = 'Folk'
GROUP BY a.nome
HAVING COUNT(*) > 5;

Se combinan las operaciones de producto cartesiano (✕) con las selecciones de unión (σ) situadas inmediatamente encima, transformándolas en operaciones de equi-join (⋈):

Se introducen proyecciones intermedias (π) destinadas a descartar todas las columnas innecesarias de las tablas, arrastrando estrictamente los atributos requeridos para las juntas y la agregación final (nome, idart, idcan, duración):
