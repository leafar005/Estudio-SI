P1 (0,5p). SQL Estándar y Diseño Relacional

Enunciado:
Explica el funcionamiento de la acción referencial on delete / on update no action. Puedes apoyarte en un ejemplo para la explicación.
🎯 Resolución:
Solución:

La cláusula NO ACTION especifica que si los cambios (de borrado o actualización) que se pretenden realizar sobre una clave primaria o única violan la integridad referencial, dicha operación no se ejecutará y el motor de la base de datos devolverá un error. Su diferencia principal con la acción RESTRICT (la cual comprueba de manera inmediata) es que NO ACTION permite que la verificación de integridad se aplace hasta el final de la transacción (mediante la opción DEFERRED), siempre y cuando la restricción haya sido configurada como aplazable.

Ejemplo de uso: > Imaginemos una base de datos con las tablas dept (departamentos) y emp (empleados), en la cual la columna deptno en emp actúa como una clave foránea que apunta a dept. Si tenemos un departamento con número 10 en dept y existe algún empleado en la tabla emp asignado a este departamento 10, cualquier intento de realizar un DELETE sobre el departamento 10, o de actualizar su identificador con un UPDATE, será abortado devolviendo un error. Esto se debe a que existen registros hijos que aún dependen de esa fila padre.

P2 (0,5p). SQL Estándar y Diseño Relacional

Enunciado:
Utilizando SQL estándar: ¿El hecho de que una vista tenga la cláusula WITH CHECK OPTION garantiza que no se pueda insertar una fila migratoria a través de ella? Justifica la respuesta.
🎯 Resolución:
Solución:

Sí, en términos predeterminados lo garantiza, pero hay excepciones dependiendo de su especificación jerárquica explícita. En el estándar SQL, la inclusión de WITH CHECK OPTION evita directamente la aparición de "tuplas migratorias", es decir, impide insertar o actualizar filas que violen la condición filtrante y desaparezcan del resultado de la vista.

Justificación detallada:

El SQL asume por defecto un alcance en cascada (CASCADED CHECK OPTION) si sólo se escribe WITH CHECK OPTION. Bajo el modelo CASCADED, el sistema comprueba tanto la condición de la propia vista como las de todas las vistas base sobre las que pudiera estar anidada, garantizando absoluta invulnerabilidad frente a filas migratorias.

Sin embargo, no lo garantiza de manera absoluta en toda la jerarquía si el desarrollador especifica explícitamente WITH LOCAL CHECK OPTION sobre una vista construida a partir de otra vista. Bajo la modalidad LOCAL, la base de datos solo comprobará la condición de la vista actual (la más superficial) de forma restrictiva. Si se insertase una fila que cumple con la restricción local, pero que no cumple con el filtro original de la vista subyacente, la inserción se permitiría temporalmente pero la fila migraría igualmente y no sería visible, rompiendo la garantía de la vista base.

P6 (0,75p). Concurrencia y Transacciones

Enunciado:
Considera un sistema con control de concurrencia multiversión. La marca de transacción será su subíndice, esto es, para cada transacción $T_i$, $M(T_i)=i$. Para el dato A tenemos las siguientes versiones:

$A_1 = 10$$\text{ ML} = 5$
$\text{ME} = 2$
$A_2 = 20$$\text{ ML} = 9$
$\text{ME} = 7$
Indica lo que pasa en cada una de las siguientes operaciones (si puede hacerse o no, qué versión se lee o escribe, etc.) y cómo queda cada una de las versiones de A.

1. $T_4$ lee A.
🎯 Resolución:
• Efecto: Como $M(T_4) = 4$, la transacción busca la versión más reciente cuya marca de escritura (ME) sea menor o igual a 4, que corresponde a $A_1$ ($ME = 2$). Dado que $M(T_4) = 4$ no supera a la marca de lectura (ML) actual de esa versión ($ML = 5$), se realiza la lectura devolviendo el valor 10, pero sin modificar las marcas, actualizándose como $\max(4, 5) = 5$.
• Estado de versiones:
- $A_1 = 10 → ME = 2, ML = 5$ - $A_2 = 20 → ME = 7, ML = 9$
1. $T_9$ lee A.
[!success]- Solución

• Efecto: Como $M(T_9) = 9$, la versión adecuada es la más reciente con $ME \le 9$, que es $A_2$ ($ME = 7$). Como $M(T_9) = 9$ se compara con el $ML(A_2) = 9$, el sistema asume $\max(9, 9) = 9$, por lo que $T_9$ lee el valor 20 y las marcas permanecen inalteradas.
• Estado de versiones:
- $A_1 = 10 → ME = 2, ML = 5$ - $A_2 = 20 → ME = 7, ML = 9$
1. $T_{13}$ escribe 13 en A.
[!success]- Solución

• Efecto: Al ser $M(T_{13}) = 13$, intenta escribir sobre la base de la versión más reciente con $ME \le 13$, que es $A_2$ ($ME = 7$). Se verifica si el dato fue leído en el "futuro" respecto a $T_{13}$ comprobando si $M(T_{13}) < ML(A_2)$; dado que $13 < 9$ es falso, se permite la escritura. $T_{13}$ crea una nueva versión $A_3 = 13$ y se le asignan las marcas $ML = 13$ y $ME = 13$.
• Estado de versiones:
- $A_1 = 10 → ME = 2, ML = 5$ - $A_2 = 20 → ME = 7, ML = 9$ - $A_3 = 13 → ME = 13, ML = 13$
1. $T_4$ escribe 25 en A.
[!success]- Solución

• Efecto: Como $M(T_4) = 4$, la versión correspondiente en su línea temporal es $A_1$ ($ME = 2$). Se evalúa si la marca temporal de $T_4$ es menor a alguna lectura realizada en el "futuro" ($ML(A_1) = 5$); como $4 < 5$ es verdadero, la operación se deniega para evitar anomalías (evitar modificar un dato que una transacción más moderna ya procesó). $T_4$ hace rollback y se reiniciará con una nueva marca temporal.
• Estado de versiones: (No cambian)
- $A_1 = 10 → ME = 2, ML = 5$ - $A_2 = 20 → ME = 7, ML = 9$ - $A_3 = 13 → ME = 13, ML = 13$
2. $T_6$ escribe 50 en A.
[!success]- Solución

• Efecto: Como $M(T_6) = 6$, busca la versión aplicable $ME \le 6$, que es $A_1$ ($ME = 2$). Se evalúa la validez con el $ML(A_1) = 5$; dado que $M(T_6) \ge ML(A_1)$ ($6 \ge 5$), se permite la operación. Se crea una nueva versión con el valor 50. Dado que esto ocurre en un instante temporal entre $A_1$ y $A_2$, se genera una versión intermedia válida (se reajustan las posiciones por marcas de escritura) con marcas $ME = 6$ y $ML = 6$.
• Estado de versiones:
- $A_1 = 10 → ME = 2, ML = 5$ - $A_{intermedia} = 50 → ME = 6, ML = 6$ - $A_2 = 20 → ME = 7, ML = 9$ - $A_3 = 13 → ME = 13, ML = 13$
